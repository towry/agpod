# Case Tool Feedback

## 0. 先断旧稿之误

前稿问题，主要有四：

1. 混了边界
   - 将 `worker`、`oracle`、`hive` 与 `case` 本体混论。
   - 然 `case` 是目标与过程管理工具；`worker` 是执行协作工具。
   - 二者可集成，但不应混为一谈。

2. 只提建议，未言收益
   - 如言“加 `test_result` 类型”，却未明说：
     - 解决何问题；
     - 省何操作；
     - 降何认知成本；
     - 对查询/交接有何改善。

3. 太抽象，少操作现场
   - 真正的痛点，不在概念本身，而在高频动作：
     - 我现在只想看当前目标与卡点，却得读一大块 case。
     - 我刚跑完测试，只想记结果并继续做事，却要手动想 record kind。
     - 我完成一步后，还得自己再切下一步。

4. 优先级不够贴真实工作流
   - 最痛的，非“未来大一统协作视图”。
   - 最痛者，其实是：
     - 看当前状态太重；
     - 高频记账太碎；
     - step 推进不顺；
     - redirect 心智负担过大。

故此稿重写时，分三层而论：

- `case` 核心能力
- `case` 与其他工具之集成面
- 优先级与实现顺序

## 1. 使用背景

此反馈基于真实长案使用，而非空想。

本轮任务特征如下：

- 目标跨多轮，非单回合可毕；
- 过程中多次切换“读代码 / 改码 / 跑测 / 冒烟 / 查日志 / 写文档”；
- 用户频繁催促“继续”“快点”“别废话”；
- 中途需记：
  - 当前目标
  - 当前 step
  - 最近关键事实
  - 哪些方向已证否
- 另有 oracle / worker / reviewer 等外部协助，但主线判断仍须落回主 agent。

故此工具之价值，不在“有无存档”，而在：

- 能否帮助 agent 快速续作；
- 能否降低记忆负担；
- 能否少打断主工作流；
- 能否在用户强压节奏下仍不失现场。

## 2. 总体判断

### 2.1 此工具已做对之处

`case` 工具最有价值之点，在于它确实提供了“长期任务的稳定锚点”。

其真正好处如下。

#### 一、Goal 固化

现状：

- 有 goal
- 有 goal constraints
- 有 success / abort 条件

好处：

- agent 不必每轮都从聊天历史重新推断“这案到底要做什么”；
- 复杂限制可固定，不致后续漂移；
- 中断后回来，能立刻复位主任务，而非重读数十轮聊天。

解决的问题：

- 长案中“目标漂移”；
- 用户先前立下的禁区被后续步骤遗忘；
- 新接手 agent 不知哪些边界不可碰。

#### 二、Current Step 可见

现状：

- `case_current` 可取当前 step。

好处：

- agent 可快速判断自己应继续“调查”“实现”“验证”何者；
- 较纯聊天上下文，当前动作感更强。

解决的问题：

- 任务做到一半被打断，再回来时不知当前卡在哪一步；
- 只知 goal，不知最近正在做哪件事。

#### 三、Fact / Finding 可沉淀

现状：

- 可记 `finding`、`evidence`、`blocker`。

好处：

- 关键结论可沉淀，而非散于聊天噪音；
- 可回看“哪些事已验证，哪些只是猜测”。

解决的问题：

- 数轮之后，重复调查已排除之方向；
- 将假设误当事实；
- 接手者需重新翻日志找关键证据。

#### 四、长案续接能力

现状：

- `case_current` + `case_resume` + 历史记录，足以做基本 handoff。

好处：

- 被中断后能恢复现场；
- 接力时不必完全依赖聊天记录。

解决的问题：

- 上下文太长；
- 多 agent 轮换时，重复消耗时间建模背景。

### 2.2 此工具当前最大问题

我认为当前主要不是“功能不足”，而是“使用负担太高”，尤其体现在高频场景。

最大三病：

1. 高频查看太重
2. 高频记录太碎
3. 方向语义太重

下文逐项展开。

## 3. Case 核心能力反馈

以下仅论 `case` 本体，不论 worker 等其他工具。

---

## 3.1 `case_current` 很有用，但仍过重

### 现状

`case_current` 会回一大块结构：

- goal
- direction
- history
- health
- last_fact
- next
- steps
- step history

### 实际痛点

在真实工作中，我最常需要的，只有五件事：

1. 当前 goal
2. 当前 active step
3. 最近 blocker / last_fact
4. 当前是否偏离目标
5. 下一步最该做什么

今若为此读取完整 case 大对象，成本偏高。

### 建议

新增 `case_brief`。

仅返回：

- `goal`
- `active_step`
- `latest_fact`
- `current_blocker`
- `next_action`
- `goal_drift_risk`

### 好处

- 降低读取成本；
- 降低 agent 每次“从大 JSON 中再抽关键信息”的二次加工；
- 更适合高频调用。

### 解决的问题

- “我只想知道现在该干什么，却得先看一大块结构”；
- 用户催促时，不想花太多 token / 注意力在 case 状态对象上；
- 长案中高频轮询状态过重。

### 代价 / 风险

- 需维护一个“摘要视图”；
- 若摘要与全量对象不同步，会造成误导。

故实现时应由同一底层状态生成，不可双写。

---

## 3.2 `case_record` 有价值，但记录粒度与分类负担太高

### 现状

可记：

- `note`
- `finding`
- `evidence`
- `blocker`

### 实际痛点

跑完一个测试后，agent 常要先想：

- 这算 evidence 还是 finding？
- 若是失败日志，是否应归 blocker？
- 若只是当前状态快照，算 note 吗？

此类判断，次数一多，心智负担甚高。

### 建议 A

为 `case_record` 增“推荐分类”能力。

例如输入内容后，由工具回：

- `recommended_kind = evidence`
- `why = contains test output and concrete runtime result`

### 好处

- 降低分类思考成本；
- 降低因人而异之记录风格；
- 使 case 数据更整齐，后续查询更可靠。

### 解决的问题

- 记录类型选择耗时；
- 不同 agent 记录风格不一，后续难检索；
- evidence / finding 混写。

### 建议 B

新增面向工程高频场景之语义化记录入口，但注意：此仍属 `case` 本体，不涉 worker。

建议新增：

- `mark_test_result`
- `mark_smoke_result`
- `mark_patch_applied`
- `mark_handoff`

### 此等建议之真正好处

#### `mark_test_result`

好处：

- 测试结果可结构化保存；
- 后续快速知道“测了什么、过没过、为何失败”；
- 交接时可直接引用，不必重述一遍。

解决的问题：

- 测试结果今多以 `evidence` 自由文本写入，格式不一；
- 难以快速筛“最后一次失败的测试是什么”。

#### `mark_smoke_result`

好处：

- 冒烟结果与普通日志分离；
- 可明确记录：
  - 入口命令
  - 关键输出
  - 成功/失败判据

解决的问题：

- smoke 往往是长案关键里程碑，但今只与普通 note 混在一起；
- 后续很难快速判断“最后一次真实 smoke 是否已通”。

#### `mark_patch_applied`

好处：

- 将“已做代码改动”与“仅分析结论”区分；
- 便于后续回顾某个失败是否已对应代码改动。

解决的问题：

- case 中常只知“认为已修”，却无结构化记录说明“何时、何文件、何意图被改”；
- 复盘时难辨“思考已做”与“代码已改”。

#### `mark_handoff`

好处：

- 可专门形成接手摘要；
- 降低中途换 agent 时的上下文重建成本。

解决的问题：

- handoff 今常混在 finding / note 中；
- 接手者需自行从多条记录中拼接主线。

### 代价 / 风险

- API 数量会增；
- 若设计过细，会反增负担。

故此类语义化入口不宜太多，只应覆盖最高频工程动作。

---

## 3.3 Step 体系有用，但推进动作不顺

### 现状

step 本身有价值，但常见动作是分散的：

- 标 done
- 找下一步
- 再设 active

### 实际痛点

工程任务中，完成一小步后，agent 最不想做的，就是再花一轮思考“下一步如何切状态”。

### 建议

新增 `step_done_and_advance`。

一次调用：

- 标当前 step 为 done
- 自动激活下一步
- 返回下一步标题与建议动作

### 好处

- 减少样板操作；
- 降低 step 维护摩擦；
- 更贴合工程推进节奏。

### 解决的问题

- “我明明已经做完这步，却还得手工再切下一步”；
- step 更新滞后，导致 case 显示与真实工作不一致。

### 代价 / 风险

- 自动推进逻辑需谨慎；
- 若下一步不唯一，可能推进错。

故应允许：

- 自动推进
- 或列出候选步让 agent 选

二者择其一。

---

## 3.4 Step 模型过于线性

### 现状

step 似假定任务大多顺序推进。

### 实际痛点

真实工程常如此：

- 一边改代码；
- 一边跑测试；
- 一边查服务日志；
- 一边整理文档；
- 一边验证 smoke。

此非纯线性，而是围绕同一主目标的并行小任务。

### 建议

`case` 核心应支持：

- 子步 `substep`
- 并行步 `parallel steps`
- 依赖关系 `blocked_by`

### 好处

- 表达真实工程状态更准确；
- 不必将并行子任务硬塞成线性顺序；
- 阻塞链条更清楚。

### 解决的问题

- 线性 step 与真实工作不符；
- 多个子验证同时进行时，case 难以如实表达。

### 代价 / 风险

- UI 与数据模型会复杂；
- 若实现过度，会让轻任务更沉。

故建议：

- 默认仍线性；
- 仅在需要时显式展开子步。

---

## 3.5 Direction / Redirect 语义太重

### 现状

当前体系中，direction 与 redirect 较强调“研究路径管理”。

### 实际痛点

工程排障时，很多情形只是：

- 沿同一路径继续深挖；
- 不是换方向；
- 更不是方案族切换。

然 agent 仍要想：

- 这算 redirect 吗？
- 要不要记 decision？
- 是否算 drift？

### 建议

将以下三者明确拆开：

1. `probe_deeper`
   - 同一路径内深化

2. `redirect`
   - 真正换方向

3. `abandon_family`
   - 正式放弃某方案族

### 好处

- 减少误用 redirect；
- 降低 agent 语义判断负担；
- 使 case 历史更清晰。

### 解决的问题

- 现很多“继续查同一问题”被迫思考成“要不要改方向”；
- redirect 过重，影响推进速度。

### 代价 / 风险

- 工具语义会略增；
- 但此增量换来的，是认知更清。

我认为值得。

---

## 3.6 Case 与 Plan 之边界不清

### 现状

今实际使用时，常有冲突：

- case 适合长期目标与证据沉淀；
- plan 适合本轮执行步骤；
- 但若二者边界不清，agent 容易犹疑。

### 实际痛点

agent 常被迫在两者中择一，而非按层次分工。

### 建议

明确定义：

- `case` 管：
  - 长期 goal
  - durable facts
  - direction history
  - current milestone

- `plan` 管：
  - 本轮执行步骤
  - 1-5 个短动作
  - 快速调整

### 好处

- 减少工具选择负担；
- 减少“到底该记 case 还是更计划”之犹疑；
- 让两类工具各司其职。

### 解决的问题

- agent 在执行层与管理层间来回摇摆；
- 任务小而急时，case 显得过重；
- 任务大而长时，plan 又不够承载历史。

### 代价 / 风险

- 需产品层面对两工具边界给出明确原则；
- 否则 agent 仍会依各自理解行事。

## 4. 操作便捷性：我最想先改的四件事

若只许先做四件，我会选：

### 4.1 `case_brief`

原因：

- 高频读取最痛；
- 最能立刻降认知成本。

### 4.2 `record_and_continue`

建议定义：

- 记一条事实
- 不改大结构
- 直接返回下一建议动作

好处：

- 贴合“刚做完一件事，顺手记下并继续”的真实节奏。

解决的问题：

- 今记录与继续是两段心智流程；
- 用户催促时尤显拖。

### 4.3 `step_done_and_advance`

原因：

- 高频；
- 直接减少样板状态维护。

### 4.4 `quick_checkpoint`

建议定义：

最少只记：

- 当前结论
- 当前卡点
- 下一步

好处：

- 特别适合用户强压节奏时保留现场；
- 可防“既没写 case，又忘了现场”。

解决的问题：

- 快速推进与完整记账之间之矛盾。

## 5. 心智负担总结

### 当前负担从何而来

不是因工具太少，而是因：

1. 高频动作不够顺手
2. 分类语义要常手动判断
3. 方向语义过重
4. 工具边界不够清

### 理想状态

理想之 `case` 工具应让我少想以下问题：

- 这条该记哪类？
- 这算不算 redirect？
- 这一步该如何切状态？
- 我现在是不是必须看完整 case？

而多把注意力放在：

- 实际证据
- 代码修复
- 测试验证
- 冒烟结果

## 6. `case` 与其他工具之集成面

此节特意与 `case` 本体分开。

`worker`、`oracle`、`reviewer` 不是 `case` 工具的一部分。
但真实工作中，它们会产生产物，需沉淀入 case。

故这里论者，不是“把 worker 做进 case”，而是“如何让 case 更好接纳外部协作产物”。

### 6.1 当前问题

现状：

- 外部工具给出结果后，主 agent 常须手工转写为 case 记录。

这会导致：

- 漏记；
- 重复摘要；
- 关键信息丢失；
- 主 agent 额外负担。

### 6.2 可选集成建议

#### 建议 A：外部结果标准化导入

例如允许：

- `import_external_result`

输入：

- source kind
- summary
- files
- conclusion

### 好处

- 保持边界清楚：
  - worker 仍是 worker
  - case 仍是 case
- 但导入动作标准化；
- 减少手工转写。

### 解决的问题

- 外部协作结果难沉淀；
- case 成为人工二次总结系统。

#### 建议 B：step 挂外部来源引用

即 step 可附：

- `related_external_runs`

而不必把外部工具本身做进 case 模型。

### 好处

- 保持解耦；
- 仍能追溯外部支持证据。

### 解决的问题

- case 与外部执行结果割裂。

## 7. `case_recall` 之使用反馈

此节专论 `case_recall` / `case_recall(context|find)`。

### 7.1 现状

就本轮真实使用而言，`case_recall` 不是主流程中的高频工具。

我更常用者是：

- `case_current`
- `case_record`
- `case_step_mark_as`

而 `case_recall` 主要只在以下场景才有明显价值：

1. 隔一段时间后续接旧案
2. 换 agent 接手
3. 一个 case 已试过多方向，要回看哪些路已证否
4. 需要回找某次具体结论，而当前上下文已不热

也就是说，`case_recall` 当前更像：

- 历史回溯工具

而非：

- 当前推进工具

此外，本轮实跑所见还有两点现状：

1. `find` 对 query 很敏感
   - query 稍偏，即可能空回；
   - 空回并不可靠地等于“无相关 case”。

2. `context` 能召回内容，但多偏“历史片段汇编”
   - 能给摘要；
   - 能给 hits；
   - 但不总能清楚区分中间推断与最终定案。

### 7.2 问题

#### 问题一：依赖手工构造 query

当前 recall 更像：

- “你先告诉我查什么，我再去找”

此模型本身无错，但在真实工程流里会带来负担：

- agent 要先形成明确检索意图；
- 要先想关键词；
- 要先决定查 repo、查 current case、还是查某个旧 case。

这一步不是零成本。

#### 问题二：命中不稳

本轮实跑里，明明存在高度相关之旧案，
但 `find` 仍可能返回空结果。

这说明：

- recall 的检索表达式与 agent 的自然提问之间，还有明显落差；
- agent 不能轻信空回；
- 往往还需换词重试。

此直接增加：

- tool call 次数；
- 检索试错成本；
- 对 recall 结果之不信任感。

#### 问题三：repo 级 context 噪声偏大

`recent_work` 会带回“最近”之事，
但最近不等于相关。

本轮实跑中，repo 级 context 混入了：

- compare/debate 文档工作
- git 状态盘点
- actions 状态检查

这些固然是近事，却不服务眼前问题。

结果是：

- recall 回来的是“近况”，不是“与当前任务最相关之历史”。

#### 问题四：不善区分结论层级

一个长案中，常有：

- 中间怀疑
- 暂时推断
- 被后续证据推翻之判断
- 最终定案

当前 recall 的摘要，容易把这些揉在一起。

此会导致：

- 接手者难一眼看出“最后到底定了什么”；
- 容易把旧推断误当现结论。

#### 问题五：不能很好抽取“当前判断最需要的历史结构”

agent 真正常需要者，不是大段历史，而是：

- 已证实什么
- 已证否什么
- 当前/最后 blocker 是什么
- 哪些文件与文档最关键

当前 recall 虽能召回材料，
却不总能结构化抽出这些关键信息。

这意味着：

- recall 有“材料价值”，
- 但对当前判断之“直接价值”仍偏弱。

#### 问题六：它是 memory / recall，不是 planner

这一点须说清。

`case_recall` 不应替 agent 决定下一步工程走向。

因为真正决定走向者，还包括：

- 当前代码状态
- 当前日志
- 当前测试结果
- 当前工作树
- 当前服务健康
- 当前用户约束

这些都不是 recall 单凭历史即可完整知道的。

故 recall 的问题，不是“不会替我做决策”，
而是：

- 没有足够低成本地把最相关历史上下文交给我。

### 7.3 期望

我对 recall 的期望，不是让它越权代决策，
而是让它更好地服务 agent 的决策。

#### 期望一：少依赖手工 query

建议提供无 query 或低 query 负担之快捷召回。

例如：

- `context_shortcut=current_step_history`
- `context_shortcut=current_blocker_history`
- `context_shortcut=recent_relevant`

其意义是：

- 不再要求 agent 先花力气设计检索表达；
- 直接围绕当前 step / blocker 召回。

#### 期望二：更强调“相关”，不只强调“最近”

repo 级 recall 不应只是 recent work 汇总，
还应更重：

- 与当前 case 的语义相关性；
- 与当前 blocker 的相似性；
- 与当前 step 的关系。

这样才能减少噪声。

#### 期望三：把历史内容按结论层级整理

理想输出不应只是片段堆叠，
而应尽量区分：

- final conclusions
- open questions
- dead ends
- superseded hypotheses

这样 agent 才不易误读。

#### 期望四：更结构化地抽取当前最值钱的信息

例如 recall 输出中，可优先抽取：

- `proven_facts`
- `dead_ends`
- `last_blocker`
- `key_files`
- `key_docs`

注意：

此仍是“提供更好历史输入”，
不是“替 agent 直接下决定”。

#### 期望五：更明确地服务接手与复盘场景

我认为 recall 最该服务好的，不是高频当前推进，
而是：

1. 冷启动续接
2. 换 agent 接手
3. 长案复盘
4. 防止重走死路

若它在这些场景中做得极顺，
其价值便已很高。

### 7.4 对 recall 的一句结论

`case_recall` 当前之局限，不在“不会告诉 agent 下一步做什么”，
而在：

- query 负担重
- 命中不稳
- repo 级噪声偏大
- 结论层级不清
- 缺少对当前判断真正有用之结构化抽取

我对它的期望也不是“替 agent 决策”，
而是：

- 更低成本地召回最相关、最干净、最可用的历史上下文，
- 让主 agent 更快、更稳地自己做判断。

## 8. `case` 与设计文档 / 实现文档之对齐问题

此点我认为非常重要，且当前缺口明显。

### 8.1 本轮实际观察

就此次实际使用而言，`case` 并无明确、一等、可靠的机制来承载：

- 本案关键设计文档是何者；
- 本案关键实现文档是何者；
- 哪份是 canonical；
- 哪份只是参考；
- 当前 case 是否已与这些文档对齐。

我在 case 中实际看到、并实际使用到者，主要仍是：

- `goal`
- `constraints`
- `direction`
- `steps`
- `last_fact`
- 若干 `record.files`

这些信息虽有用，但并不能替代“文档锚点”。

### 8.2 这会带来什么问题

#### 问题一：接手者不知先读何文

现状下，case 只能告诉我“要做什么”，不能可靠告诉我：

- 进入此案前，必须先读哪份设计文；
- 哪份实现文是当前版本；
- 哪份 runbook 已过时。

结果是：

- 新 agent 仍要靠聊天、零散文件、人工猜测来定入口。

#### 问题二：文档版本易漂

长案推进中，常有如下变化：

- 设计文修订了；
- 实现文补充了；
- runbook 已过时；
- 某份文档被新文替代。

若 case 不显式绑定文档，则会出现：

- case 仍按旧理解推进；
- agent 不知自己依据的是哪一版文档；
- 文档虽改，任务认知却未同步。

#### 问题三：事实与文档割裂

本轮里，我能在 case 中记很多 finding / evidence。
但 case 并不能清楚告诉我：

- 这些发现应回写到哪份长期文档；
- 哪份文档应随当前结论更新；
- 哪份文档只是会话中临时参考，不应作为 canonical。

结果是：

- case 与长期文档体系是松散耦合；
- 任务推进与文档维护容易脱节。

### 8.3 为什么这会增加 agent 心智负担

因为 agent 不得不额外自行维护三种映射：

1. 当前 case 对应哪些关键文档
2. 这些文档中哪份是主、哪份是辅
3. 当前 step 之前应先读哪份

此种映射，今不在工具里，便只能放在 agent 脑中或聊天中。
这正是应由工具替 agent 承担之负担。

### 8.4 我建议新增“文档锚点”

建议于 case 层加入一等字段，例如：

- `design_docs`
- `implementation_docs`
- `runbook_docs`

每项至少含：

- `path`
- `role`
- `required`
- `status`
- `notes`

建议语义如下。

#### `path`

文档路径。

好处：

- 让 case 明确指向具体文档，而非让 agent 事后回忆。

解决的问题：

- “到底是哪份文？”

#### `role`

建议至少支持：

- `canonical`
- `reference`
- `draft`
- `superseded`

好处：

- 明确主次；
- 避免把临时文档当正式依据。

解决的问题：

- 多份文并存时，agent 不知以谁为准。

#### `required`

表示是否进入本案或某步前必须先读。

好处：

- 可将“先读文档”从隐性经验改成显式规则。

解决的问题：

- 接手 agent 不知先读何文；
- 工具无法提醒文档前置条件。

#### `status`

建议至少支持：

- `active`
- `stale`
- `superseded`

好处：

- 文档版本变化可被 case 感知。

解决的问题：

- runbook 已漂移，但 case 仍引用旧文；
- 设计文已换版，agent 却继续按旧版执行。

#### `notes`

简单说明此文与本案之关系。

好处：

- 减少接手者自行猜测文档用途。

解决的问题：

- 文档虽列出，然不知其与本案之具体关系。

### 8.5 仅有 case 级绑定还不够，还应有 step 级绑定

我建议不仅 case 顶层要有文档锚点，step 亦应可附文档需求。

例如：

- `required_docs`

意义是：

- 做此 step 前，应先读哪些文；
- 此 step 的判断依据来自哪些文。

### 这样做的好处

#### 好处一：进入某步前，工具可直接提醒

例如：

- “当前 step 需要先阅读 2 份文档”

这样 agent 无须再靠记忆或聊天补全。

#### 好处二：文档与动作真正对齐

不仅知道“本案参考了哪些文”，
还知道“当前这一步具体依赖哪些文”。

#### 好处三：接手更稳

新 agent 不止知道本案有哪些文档，
还能知道：

- 眼下这一步先看哪份
- 哪些文只是背景

### 8.6 我认为此改进之价值很高

此改进的价值，在于它直接减少以下高成本行为：

- 靠聊天回忆文档入口；
- 靠手工判断 canonical 文档；
- 靠 agent 自己记忆哪些文已漂移；
- 靠人工再决定哪份文该回写。

也就是说，它不是“文档管理装饰功能”，而是：

- 减少接手成本；
- 减少误引用旧文；
- 减少实现与设计脱节；
- 减少长期文档与 case 事实割裂。

### 8.7 我建议的最小可行实现

若只做最小版，我建议先上两层：

#### 第一层：case 级文档锚点

支持：

- `design_docs`
- `implementation_docs`

每项仅含：

- `path`
- `role`
- `required`

先不追求最全。

#### 第二层：step 级 `required_docs`

只需让当前 step 可挂一组路径即可。

这样已能解决大半现实问题：

- 进入本案先读何文；
- 当前这一步先读何文。

### 8.8 一句结论

当前 `case` 工具与设计文档 / 实现文档之对齐，基本是缺失的。

此缺失之坏处，不在“文档列表不够漂亮”，而在：

- agent 必须自己维护文档映射；
- case 难以成为真正可接手、可执行、可长期维护之任务中枢。

故我认为，“文档锚点”应是一个很值得做的 case 优化项。

## 9. 优先级建议

### P0：立刻改善高频体验

1. `case_brief`
2. `record_and_continue`
3. `step_done_and_advance`
4. `quick_checkpoint`

理由：

- 不改这些，agent 每日都痛；
- 改了这些，立刻顺滑。

### P1：降低语义负担

1. 分类推荐
2. `probe_deeper / redirect / abandon_family` 分拆
3. case / plan 边界明确

理由：

- 这些不改，agent 仍会频繁犹疑。

### P2：更贴近真实工程流

1. 子步
2. 并行步
3. blocked-by

理由：

- 这将提升对真实任务结构之表达力。

### P3：外部集成

1. 外部结果标准导入
2. step 关联外部来源

理由：

- 有价值，但不是第一痛点；
- 应在 case 本体顺手之后再做。

### P4：文档对齐

1. case 级 `design_docs`
2. case 级 `implementation_docs`
3. step 级 `required_docs`

理由：

- 此非最高频操作优化；
- 但对长案接手、避免误读旧文、长期维护极重要。

## 10. 一句结论

我对 `case` 工具之最终评价是：

它的方向是对的，价值也真实存在；但若要从“能用”变“真正在高压工程流中常用”，必须先优化高频动作之便捷性，而不是先继续堆概念或扩字段。

最该优先解决者，不是“功能是否更大”，而是：

- 看状态更轻；
- 记事实更顺；
- 推 step 更快；
- 少为语义分类分心。

若此四者得正，则 `case` 工具会由“任务档案”进为“任务驾驶舱”。
